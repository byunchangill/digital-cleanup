# Sortmate 루프 회전 프롬프트

너는 Sortmate 앱 무인 빌드 루프의 1회전을 실행한다. 이 프롬프트는 ralph-loop에 의해 매 회전 동일하게 주입된다. 상태는 오직 `_workspace/` 아래 파일로만 이어진다 — 이전 회전의 대화 내용을 안다고 가정하지 마라.

## 회전 절차

1. **상태 파악**: `_workspace/PROGRESS.md`와 `_workspace/LOOP_LOG.md`를 읽는다.
   - `_workspace/`가 없으면 초기 실행이다 → 이번 회전은 스캐폴딩(`backend/`, `frontend/` 초기 생성)부터 시작해 auth 모듈을 진행한다.
2. **이번 회전의 작업 1건 결정** (우선순위 순):
   1. QA 실패 항목이 잔존하고 실패 횟수가 2회 미만인 모듈 → 그 모듈의 수정+QA 재검증 (회귀 재검증 최우선 원칙)
   2. 미착수/미완료 모듈 중 의존 순서상 첫 번째: auth → item → home → cleanup → vault → my
   - admin 모듈은 2차 범위이므로 절대 착수하지 않는다.
3. **실행**: `sortmate-build` 스킬을 호출하되, SKILL.md의 **"루프 모드"** 절 규칙을 적용한다 — 이번 회전에는 결정한 모듈 **1개만** 처리하고, 사용자에게 질문하지 않는다.
4. **기록**: 회전 종료 시(성공이든 실패든 반드시):
   - `_workspace/PROGRESS.md` 갱신 (모듈×단계 상태 + 실패 횟수)
   - `_workspace/LOOP_LOG.md`에 1줄 추가: `| 회전번호 | 모듈 | 도달 단계 | 결과 | 특이사항 |` (파일이 없으면 표 헤더와 함께 생성)

## 종료 조건

6개 모듈(auth, item, home, cleanup, vault, my)이 **모두** `QA통과` 또는 `실패(2회)` 상태이면:

1. `_workspace/LOOP_LOG.md` 하단에 최종 요약을 쓴다: 통과 모듈, 실패 모듈과 사유, `[가정]`/`[자율결정]` 목록, 사용자 확인 필요 항목.
2. 응답 마지막에 정확히 이 태그를 출력한다: `<promise>SORTMATE_LOOP_COMPLETE</promise>`

빌드 환경 자체가 불능(Java/Node/gradlew/npm 미설치 또는 실행 불가)이라 어떤 모듈도 진행할 수 없는 경우에도, 사유를 LOOP_LOG.md에 기록하고 위 완료 문구를 출력하여 루프를 종료한다.

종료 조건에 해당하지 않으면 완료 문구를 절대 출력하지 마라 — 다음 회전이 이어서 실행된다.
