# Sortmate 약관·개인정보 처리방침 출시 체크리스트

> 기준일: 2026-07-19  
> 상태: 공개 전 필수 확인사항  
> 대상 문서: `terms-of-service.md`, `privacy-policy.md`

이 체크리스트는 공개용 문서에 남아 있는 가정과 현재 구현의 충돌을 정리합니다. 체크가 끝나기 전에는 두 문서를 운영 환경에 그대로 게시하지 마세요.

## 1. 사업자·담당자 정보

- [ ] 상호/법인명, 대표자, 사업자등록번호, 주소, 전화번호를 입력한다.
- [ ] 유료서비스를 판매하면 통신판매업 신고번호와 신고기관을 입력한다.
- [ ] 개인정보 보호책임자 또는 대표자, 담당부서 및 권리행사 연락처를 확정한다.
- [ ] `support@sortmate.app` 메일 수신과 처리 절차를 실제로 운영한다.
- [ ] 공고일·시행일·문서 버전을 확정하고 이전 버전 보관 경로를 만든다.

## 2. 실제 데이터 흐름 확정

- [ ] 운영 데이터베이스, 파일 저장소, CDN, 백업, 이메일, AI, 분석, 고객지원, 결제 사업자를 전수 조사한다.
- [ ] 각 수탁자와 위탁 업무, 처리 항목, 보관기간, 재위탁 여부를 개인정보 처리방침 제4조에 반영한다.
- [ ] 구글 로그인과 해외 클라우드·AI 서비스의 국외 이전 여부를 확인하고 이전받는 법인, 국가, 연락처, 목적, 항목, 시기·방법, 기간, 법적 근거 및 거부 효과를 제5조에 채운다.
- [ ] 운영 로그의 실제 항목(IP, User-Agent, 기기 식별자 등)과 보관기간을 확정한다.
- [ ] 알림, 내보내기 파일, 상담 기록, 백업본의 구체적인 보관기간을 확정한다.
- [ ] 실제로 처리하지 않는 항목과 도입하지 않은 수탁자 행은 공개본에서 삭제한다.

## 3. 동의와 가입 흐름

- [ ] 현재 이메일 가입의 `agreedToTerms` 하나로 이용약관과 개인정보 처리방침을 묶어 동의받지 않는다. 약관 동의와 개인정보 관련 선택 동의를 목적별로 분리한다.
- [ ] 서비스 제공에 꼭 필요한 개인정보는 계약 이행 근거와 필수 안내를 명확히 하고, AI 학습·사용 통계·맞춤형 제안은 각각 선택 동의로 구성한다.
- [ ] 선택 동의는 기본값을 `false`로 한다. 현재 `PrivacySettings.aiTrainingConsent`와 `personalizedSuggestions` 기본값은 `true`이므로 출시 전에 수정한다.
- [ ] 동의 문구, 문서 버전, 동의 일시, 동의 철회 일시를 서버에서 증빙할 수 있도록 기록한다. 현재 회원 엔티티에는 약관 동의 이력이 저장되지 않는다.
- [ ] 소셜 로그인 신규 가입에도 약관 확인과 필요한 동의 단계를 추가한다. 현재 소셜 인증 성공 시 동의 기록 없이 계정이 생성될 수 있다.
- [ ] `LoginPage.jsx`와 `EmailAuthPage.jsx`의 `#` 링크를 실제 약관·처리방침 화면 또는 URL로 연결한다.
- [ ] 만 14세 미만 가입 제한을 UI·API에서 집행하거나, 아동 가입을 허용할 경우 법정대리인 동의 절차를 별도로 구현한다.

## 4. AI와 민감한 이용자 콘텐츠

- [ ] "익명 데이터"가 법적으로 재식별 불가능한 익명정보인지 검증한다. 그렇지 않으면 가명정보 또는 개인정보로 정확히 표현한다.
- [ ] AI 학습용 데이터의 출처, 포함 항목, 가명·익명처리 방법, 학습 반영 후 철회 가능 범위, 보관기간을 확정한다.
- [ ] 외부 AI 사업자에게 원문·이미지를 보내는지 확인하고 위탁·국외 이전 및 모델 학습 금지 설정을 계약과 정책에 반영한다.
- [ ] 신분증, 금융, 건강 정보가 담긴 스크린샷을 분석·보관할 계획이면 민감정보·고유식별정보 처리의 필요성, 별도 동의, 마스킹, 접근통제 및 보관기간을 법률 전문가와 검토한다.
- [ ] 자동 삭제는 반드시 대상, 기준, 복구 가능 기간과 이용자 확인 절차를 명확히 한다.

## 5. 보안·삭제 구현과 표시 문구

- [ ] `PrivacyControlsPage.jsx`의 "기기를 떠나기 전 항상 AES-256으로 암호화" 문구는 현재 코드에서 검증되지 않는다. 종단간/클라이언트 암호화를 실제 구현·검증하거나 문구를 삭제한다.
- [ ] `PlanPage.jsx`의 "ISO 인증" 표시는 실제 인증서가 없으면 삭제한다.
- [ ] `Plan`의 "고급 보안 프로토콜", "24/7 우선 지원"은 실제 제공 체계를 마련하거나 표시에서 삭제한다.
- [ ] 탈퇴 API는 현재 30일 뒤 파기 예정 레코드만 생성한다. 실제 계정·콘텐츠·토큰·설정·알림·내보내기·구독 데이터 파기 배치와 실패 재처리, 감사 로그를 구현한다.
- [ ] 탈퇴 유예 중 재로그인이 실제로 탈퇴를 취소하는지 확인하고, 미구현이면 화면·약관의 해당 문구를 수정한다.
- [ ] 법정 보존 자료는 일반 계정 데이터와 논리·물리적으로 분리하고 목적 외 접근을 막는다.
- [ ] 파일 삭제, 휴지통, 백업 순환과 복구 가능 기간을 제품 전반에서 같은 용어와 기간으로 표시한다.
- [ ] 브라우저 `localStorage`에 저장하는 액세스·리프레시 토큰의 XSS 위험을 검토하고 HttpOnly·Secure·SameSite 쿠키 등 더 안전한 세션 설계를 평가한다.
- [ ] 운영 환경의 TLS, 저장 암호화, 키 관리, 접근권한, 접속기록, 백업 및 침해사고 대응 절차를 검증한 뒤 공개 문구를 확정한다.

## 6. 결제·환불

- [ ] 현재 프리미엄 업그레이드와 구매 복원은 모의 기능이다. 실제 결제 전까지 결제가 일어난 것으로 오인시키지 않는다.
- [ ] 실제 판매 주체, 가격·부가세, 결제주기, 자동갱신, 무료체험 전환, 해지 경로, 환불 주체 및 처리기한을 확정한다.
- [ ] 디지털콘텐츠 제공 개시로 청약철회가 제한되는 경우 결제 전에 제한 사유를 별도 고지하고 명시적 동의를 받는다.
- [ ] 앱마켓 결제와 웹 결제의 해지·환불 절차 차이를 구매 화면과 약관에 반영한다.
- [ ] 가격 인상 또는 무료에서 유료 정기결제로 전환할 때 시행 시점의 전자상거래법상 사전 동의·고지 요건을 다시 확인한다.

## 7. 공개와 운영

- [ ] 회원가입 전 약관과 개인정보 처리방침 전문을 열람·저장할 수 있게 한다.
- [ ] 개인정보 처리방침은 홈페이지와 앱에서 항상 쉽게 찾을 수 있게 게시한다.
- [ ] 중요 변경 30일 전, 일반 변경 7일 전 공지 절차와 개별 통지 채널을 마련한다.
- [ ] 개인정보 열람·전송·정정·삭제·처리정지·동의 철회 요청의 본인확인, 접수, 처리, 거절 사유 통지 절차를 마련한다.
- [ ] 개인정보 유출 대응, 이용자 통지 및 관계기관 신고 절차를 문서화하고 훈련한다.
- [ ] 공개 직전 국내 개인정보·전자상거래 전문 변호사 또는 개인정보보호 담당자의 최종 검토를 받는다.
