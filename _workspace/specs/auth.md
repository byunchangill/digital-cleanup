## 인증 토큰 체계 (JWT) — vault PIN과의 경계 정의
Sortmate 인증은 **2계층**으로 구성된다. 두 계층을 혼동하면 통합 시 토큰 충돌이 발생하므로 여기서 확정한다.

1. **1계층 — 계정 인증 (auth 모듈 소관, 본 명세):**
   - 로그인(소셜/이메일) 성공 시 서버가 **JWT accessToken + refreshToken** 발급.
   - `accessToken`: JWT, 만료 30분(`[가정]`), 모든 보호 API에 `Authorization: Bearer {accessToken}` 헤더로 전달.
   - `refreshToken`: 만료 14일(`[가정]`), accessToken 재발급에만 사용. 갱신 엔드포인트로만 교환.
   - JWT 클레임(`[가정]`): `sub`(userId), `email`, `provider`, `iat`, `exp`.
2. **2계층 — 앱 잠금 PIN (vault 모듈 소관, 본 명세 제외):**
   - PIN은 **로그인된 기기에서 vault 접근을 다시 잠그는 로컬 2차 인증**이며, JWT를 대체하지 않는다.
   - PIN 검증 결과는 JWT를 재발급하지 않고 클라이언트 세션의 vault-unlock 플래그로만 관리한다(`[가정]` — vault 모듈 확정 대상).
   - **경계 규칙:** auth 계약의 어떤 엔드포인트도 PIN을 요청/응답 필드로 포함하지 않는다. auth는 계정 소유권만 검증하고, PIN은 vault가 별도 검증한다.

## 사용자 흐름
```
[로그인 /login]
  ├─ 카카오/구글/애플 버튼 → 소셜 OAuth 인증 → (성공) 홈/보관함  ※홈은 auth 모듈 밖
  ├─ "이메일로 로그인" 버튼 → [이메일 로그인/회원가입 /auth/email]
  └─ (비밀번호 분실 진입) → [계정 복구 /recovery] 또는 [비밀번호 재설정 /password/reset]

[이메일 로그인 /login/email (com_004_1, 한글 로그인 전용)]
  ├─ 이메일+비밀번호(표시 토글) → [로그인] → AUTH-02 → (성공) 홈/보관함
  ├─ "비밀번호 찾기" → [비밀번호 재설정 /password/reset] (AUTH-04)
  ├─ 하단 "계정이 없으신가요? 회원가입" → [이메일 회원가입 /signup]
  ├─ 소셜: 카카오 / 구글
  └─ arrow_back → 이전 화면(history.back)

[이메일 회원가입 /signup (com_004, 한글 가입 전용)]
  ├─ 이메일 + 비밀번호(표시 토글) + 비밀번호 확인 → [회원가입 하기] → AUTH-08 → (성공) 자동 로그인 → 홈 [가정]
  ├─ 약관 체크박스 "이용 약관 및 개인정보 처리방침에 동의합니다" (필수)
  │    └─ "이용 약관"/"개인정보 처리방침" 링크 → [약관 및 정책 /legal] (AUTH-09)
  ├─ "또는 다음으로 시작하기" 구글/카카오 → 소셜 OAuth (AUTH-01)
  └─ 하단 "이미 계정이 있으신가요? 로그인" → [이메일 로그인 /login/email]

[약관 및 정책 /legal (com_005)]
  ├─ 탭: [이용 약관] / [개인정보 처리방침] 전환 (조문 카드 리스트)
  ├─ 콘텐츠 소스: docs/legal/terms-of-service.md · privacy-policy.md 전문 [가정]
  ├─ 진입: 회원가입 폼 TOS 링크(AUTH-08) 또는 마이/설정 → 약관 보기
  ├─ [확인] → 이전 화면 복귀. 회원가입 맥락이면 동의 체크 유도(열람 완료 신호) [가정]
  └─ arrow_back → history.back

[계정 복구 /recovery (com_008)]
  ├─ "이메일로 인증" → 매직 링크 발송 → (토스트) → 이메일 링크 클릭 → [새 비밀번호 설정]
  ├─ "복구 코드로 인증" → 24자리 백업 키 입력 화면 → (검증 성공) → [새 비밀번호 설정]
  ├─ "고객 센터 문의" → 고객 지원 포털  ※외부/off-module
  └─ 뒤로가기(arrow_back) → 이전 화면(history.back)

[비밀번호 재설정 /password/reset]
  ├─ 이메일 입력 → "재설정 링크 발송" → (로딩→성공 토스트) 이메일로 링크 발송
  └─ "로그인으로 돌아가기" → [로그인 /login]

[새 비밀번호 설정 /password/new (com_007)]
  ├─ 진입 조건: 유효한 재설정/복구 토큰 보유 (링크 클릭 또는 복구코드 검증 통과)
  ├─ 진행 표시: 3단계 중 3번째 활성 → (복구방법선택 → 인증 → 새 비번설정) 흐름 암시 [가정]
  ├─ 새 비밀번호 + 확인 입력 → "비밀번호 변경" → (성공 토스트) → [로그인]으로 복귀 [가정]
  └─ 뒤로(arrow_back) → history.back
```

### email_login_sign_up_com_004 — 이메일 회원가입 (한글 가입 전용)
> 파일 교체 이력: 초기 영문 LOGIN/SIGN UP 탭 통합형 → **한글 회원가입 전용 화면**으로 교체(2026-07-20 확인). 로그인은 별도 화면(com_004_1)으로 완전 분리.
- **표시 데이터:** 브랜드명("Sortmate"), 헤드라인("새로운 시작")+부제, 소셜 구분선("또는 다음으로 시작하기").
- **입력 요소:** 이메일 `input[type=email]`, 비밀번호 `input[type=password]`(placeholder "8자 이상 입력해주세요") + 표시 토글(visibility_off), **비밀번호 확인 `input[type=password]`(id=password_confirm)**, 약관동의 체크박스(id=terms), [회원가입 하기] submit, 소셜 구글·카카오 버튼, 하단 "로그인" 링크.
- **상태 변형:** submit 시 버튼 "처리 중..."(disabled+opacity). 에러/로딩 상태 상세는 화면 미표현 → 표준 에러.
- **이동 경로:** 회원가입 하기 → AUTH-08. 약관/개인정보 링크 → `/legal`(AUTH-09, terms_and_policies_com_005). 소셜 → AUTH-01. 하단 로그인 → `/login/email`(com_004_1). 성공 시 홈(off-module).
- **[가정] 자동 로그인:** 가입 성공 후 별도 로그인 화면 복귀 UI가 없으므로, 가입 성공 시 **AUTH-02와 동일한 토큰 봉투를 발급해 즉시 로그인**(기존 결정 유지).
- **[가정] 약관동의:** 체크박스는 가입 필수. 클라이언트 submit 차단이 1차, 서버도 `agreedToTerms=true` 요구(신뢰 경계 이중 검증). 체크박스 내 "이용 약관"/"개인정보 처리방침"은 `/legal` 링크로, 열람 후 동의 체크가 UX 흐름.
- **[상충] 비밀번호 정책 (8자 vs 12자):**
  - 화면 근거: placeholder "8자 이상 입력해주세요" → 최소 8자만 안내.
  - 기존 계약/구현: `PasswordPolicyValidator` = 최소 12자 + 대문자 1개 + 특수문자 1개 + 흔한 패턴 금지(set_new_password 정책, AUTH-05).
  - **권고안(채택):** 기존 12자 정책을 **유지**하고, 화면 placeholder 문구를 정책에 맞게 수정("12자 이상, 대문자·특수문자 포함")하도록 frontend-dev에 요청. 근거: 재설정(AUTH-05)과 가입 정책이 갈리면 사용자가 재설정 시 혼란·검증 불일치. placeholder는 참고 표현이므로 정책 소스가 아니라 정책에 종속시키는 것이 안전.
  - 대안(비채택): 가입만 8자로 완화 → 재설정 정책과 불일치·보안 하향이라 지양.
- **[가정] passwordConfirm = 클라이언트 검증만:** 비밀번호 확인 필드는 **입력 오타 방지용 클라이언트 검증**(password === passwordConfirm)으로 충분. 서버는 확정된 `password` 하나만 받으면 되므로 **AUTH-08 계약에 `passwordConfirm` 필드를 추가하지 않는다**. 근거: AUTH-05(새 비밀번호)는 두 값을 서버로 보내 `PASSWORD_MISMATCH`를 두었으나, 가입 화면은 토큰 없는 단순 폼이고 확인 불일치는 제출 전 클라이언트에서 100% 잡힌다 → 서버 필드는 불필요(YAGNI). 일관성을 위해 서버 필드를 원하면 추가 가능하나 계약 변경으로 취급.

### email_login_com_004_1 — 이메일 로그인 (한글 전용)
- **표시 데이터:** 뒤로가기 + "이메일 로그인" 헤더, "반가워요!" 헤드라인, 부제("다시 만나서 기뻐요. 계정 정보를 입력해주세요."), "or" 소셜 구분선.
- **입력 요소:** 이메일 `input[type=email]`, 비밀번호 `input[type=password]` + 표시 토글(visibility), "비밀번호 찾기" 링크, [로그인] submit, 하단 "회원가입" 링크, 소셜 버튼 **카카오·구글**(2종).
- **상태 변형:** 에러/로딩 화면 미표현 → 표준 에러.
- **이동 경로:** 로그인 → AUTH-02(경로 `/api/auth/login` 확정). 비밀번호 찾기 → `/password/reset`(AUTH-04). 하단 회원가입 링크 → `/signup`(com_004, AUTH-08). 소셜 → AUTH-01.
- **화면 구조 = 로그인/회원가입 분리 2화면(확정):** 004_1(로그인)과 004(회원가입)는 **각각 독립 화면**이며 서로 하단 링크로 이동. 초기 004 영문 탭 통합형은 폐기됨. 두 화면 모두 한글, 소셜 구글+카카오로 일관. 기존 EmailAuthPage(탭형)를 구현했다면 **로그인/회원가입 2개 페이지로 재구성** 필요(frontend-dev). 라우트: 로그인 `/login/email`, 회원가입 `/signup`. 동일 엔드포인트(AUTH-02/08) 재사용 → **계약 변경 없음**.

### terms_and_policies_com_005 — 약관 및 정책
- **표시 데이터:** 뒤로가기 + "약관 및 정책" 헤더, 탭([이용 약관]/[개인정보 처리방침]), 조문 카드 리스트(제목+본문), 하단 [확인] 버튼.
- **입력 요소:** 탭 전환 버튼 2종, [확인] 버튼, arrow_back.
- **상태 변형:** 탭 전환 시 해당 섹션 표시(클라이언트 토글). 로딩/에러 없음(정적 콘텐츠).
- **[가정] 콘텐츠 소스 = docs/legal 전문:** 화면 HTML의 조문(제1~5조, 개인정보 1~4항)은 **요약 발췌**이며, 실제 서빙 콘텐츠는 `docs/legal/terms-of-service.md`(제1~16조+부칙)·`docs/legal/privacy-policy.md`(1~13절)의 **전문**을 사용한다. 근거: 법적 문서는 전문 노출이 원칙이고, 화면 발췌본과 전문이 다르면 전문이 우선(`[상충]` 시 전문 채택).
- **[가정] 서빙 방식 = 프론트 정적 포함:** 약관/방침 md를 프론트 번들에 정적 포함(빌드 시 import)하여 렌더. **백엔드 API 불필요** — AI stub들과 동일하게 최소화. 버전 관리가 계약상 필요해지면(동의 시점 버전 기록 등) 아래 "약관 버전 관리" 참조.
- **[가정] 약관 버전 관리:** 현 단계는 정적 서빙이므로 버전은 문서 헤더의 `v0.9`/시행일 문자열로 충분. **AUTH-08 회원가입 시 동의한 약관 버전을 서버가 기록할 필요가 생기면**, signup 요청에 `termsVersion`(선택) 필드를 추가하는 방식으로 확장 가능(현재는 불필요, YAGNI). 계약에 확장 지점만 명시하고 필드는 추가하지 않는다.
- **이동 경로:** 확인/뒤로 → 이전 화면. 회원가입 폼에서 진입 시 열람 후 TOS 체크 유도(AUTH-08 `agreedToTerms`와 연결, **signup 계약 변경 없음**).

### reset_password — 비밀번호 재설정 요청
- **표시 데이터:** 안내 문구, 성공 토스트 문구("링크가 발송되었습니다! 편지함을 확인해 주세요.").
- **입력 요소:** 이메일 `input[type=email]`(id=email, placeholder "예: name@company.com"), [재설정 링크 발송] submit 버튼.
- **상태 변형:** 기본 / 로딩("전송 중...", 버튼 disabled + 스피너) / 성공("성공적으로 발송됨" + 토스트). 실패 상태는 화면 미표현.
- **이동 경로:** [로그인으로 돌아가기] → `login.html`(`/login`).
- **`[자율결정]`:** 존재하지 않는 이메일이어도 성공 토스트를 노출하는 것이 화면 동작(1.5초 후 무조건 성공). 계정 열거(enumeration) 방지를 위해 서버도 **항상 200 성공 봉투**를 반환하도록 계약 확정. 근거: 화면이 이메일 유효성에 따른 분기 UI를 전혀 제공하지 않음.

### set_new_password_com_007 — 새 비밀번호 설정
- **표시 데이터:** 진행 트래커(3단계, 3번째 활성), 보안 강도 라벨("강함")+게이지, 비밀번호 가이드라인 2줄.
- **입력 요소:** 새 비밀번호 `input[type=password]`(id=password, placeholder "최소 12자"), 비밀번호 확인 `input[type=password]`(id=confirm-password), 각 입력의 표시/숨김 토글 버튼, [비밀번호 변경] submit 버튼.
- **비밀번호 정책(화면 근거):** 최소 12자 / 최소 1개 대문자 + 1개 특수문자 / "123"·"password" 등 흔한 패턴 금지.
- **상태 변형:** 기본 / 성공 토스트("비밀번호가 성공적으로 업데이트되었습니다."). 강도 미터는 클라이언트 표시용. 불일치·정책 위반 에러는 화면 미표현 → 계약 에러 코드로 정의.
- **이동 경로:** 뒤로(history.back). 성공 후 이동 화면은 미명시 → `/login` 복귀로 `[가정]`.
- **진입 전제:** 유효한 `resetToken`(AUTH-04 링크) 또는 `recoveryToken`(AUTH-07 검증) 보유 상태.

### account_recovery_com_008 — 계정 복구 방법 선택
- **표시 데이터:** 안내 문구, 옵션 카드 2종(이메일 인증 "수신함으로 매직 링크를 보내드립니다" / 복구 코드 "24자리 백업 키를 입력하세요"), 하단 "고객 센터 문의", 빌드 태그("COM-008 • Sortmate Secure Auth v2.4"), 복구 성공 토스트("이메일로 복구 링크를 보냈습니다.").
- **입력 요소:** [이메일로 인증] 버튼, [복구 코드로 인증] 버튼, [고객 센터 문의] 버튼.
- **상태 변형:** 각 버튼 클릭 시 토스트. 자체 폼 없음(선택 후 다음 화면으로 위임).
- **이동 경로:** 이메일로 인증 → 매직 링크 발송(AUTH-06). 복구 코드로 인증 → 24자리 입력 화면(화면 미제공, `[가정]`) → 검증(AUTH-07) → 새 비밀번호 설정. 고객 센터 → 외부 포털. 뒤로 → history.back.

## 데이터 모델 초안
> 백엔드 참고용 초안. 최종 스키마는 backend-dev 재량.

- **User**
  - `id` (PK), `email` (unique), `displayName`, `provider` (enum: `EMAIL`|`KAKAO`|`GOOGLE`|`APPLE`), `providerId` (소셜 고유 ID, nullable), `passwordHash` (provider=EMAIL만), `createdAt`, `updatedAt`.
- **PasswordResetToken** (AUTH-04/05용)
  - `id`, `userId` (FK), `token` (URL-safe, 발송 링크에 포함), `expiresAt`, `usedAt` (nullable), `createdAt`. TTL `[가정]` 30분.
- **RecoveryCode** (AUTH-07용)
  - `id`, `userId` (FK), `codeHash` (24자리 백업 키 해시), `usedAt` (nullable), `createdAt`. 1회용. 24자리 형식 `[가정]` = 영숫자 24문자(4-4-4-4-4-4 그룹핑 표시).
- **RefreshToken** (AUTH-03용, `[가정]`)
  - `id`, `userId` (FK), `token`, `expiresAt`, `revokedAt` (nullable).

## 가정 및 상충
- `[해소] AUTH-02` 이메일 로그인 입력 폼 화면(com_004) 확보됨 → 필드(email/password) 확정. 이전 `[가정]` 해소.
- `[가정] AUTH-08` 회원가입 성공 시 자동 로그인(AUTH-02 동일 토큰 봉투 발급). 근거: 화면에 가입→로그인 중간 단계 없음.
- `[가정] AUTH-08` TOS 동의(`agreedToTerms`) 서버 필수 검증. 회원가입 비밀번호는 기존 PasswordPolicyValidator 규칙 재사용.
- `[확정] AUTH-02/08` 로그인(com_004_1)/회원가입(com_004) **분리 2화면**. 초기 004 영문 탭형 폐기. 라우트 `/login/email`, `/signup`. 기존 탭형 EmailAuthPage는 2페이지로 재구성. 동일 API, 계약 무변경.
- `[상충] AUTH-08` 비밀번호 정책: 화면 placeholder "8자 이상" vs 기존 정책 12자+대문자+특수문자. **권고: 12자 정책 유지 + placeholder 문구 수정.** 재설정(AUTH-05)과 정책 통일 위함.
- `[가정] AUTH-08` passwordConfirm은 클라이언트 검증만(오타 방지). 서버는 `password` 하나만 수신, AUTH-08 계약 필드 무추가(YAGNI).
- `[가정] AUTH-09` 약관/정책 콘텐츠는 docs/legal 전문을 소스로 프론트 정적 서빙. 백엔드 API 불필요. 화면 발췌본과 전문 상충 시 전문 우선.
- `[가정] AUTH-09` 약관 버전은 정적 문자열(v0.9). 동의 버전 서버 기록이 필요해지면 signup에 `termsVersion` 선택 필드 확장(현재 미추가, YAGNI).
- `[가정] AUTH-03` 토큰 갱신/로그아웃 화면은 없으나 JWT 체계상 필수 인프라로 포함.
- `[가정]` accessToken 30분 / refreshToken 14일 / resetToken 30분 TTL — 화면 근거 없음. 보안 기본값으로 설정.
- `[가정]` 복구 코드 "24자리" 형식을 영숫자 24문자로 해석(화면 문구 "24자리 백업 키"만 존재).
- `[가정]` set_new_password 성공 후 `/login`으로 복귀(화면에 다음 라우트 미명시).
- `[가정]` 진행 트래커 3단계 = 복구방법선택 → 인증 → 새 비번설정 흐름.
- `[자율결정] AUTH-04` 계정 존재 여부와 무관하게 항상 성공 봉투 반환(계정 열거 방지). 근거: reset_password 화면이 이메일 유효성 분기 UI를 전혀 갖지 않고 무조건 성공 처리함.
- `[자율결정]` reset_password(단순 이메일 링크)와 account_recovery의 "이메일로 인증"(매직 링크)이 기능적으로 중복. 두 화면 모두 유지하되 **동일 백엔드 엔드포인트(AUTH-04/06를 하나로)로 볼 여지**가 있음 → 진입 맥락이 다르므로(재설정 vs 복구 허브) 계약상 별도 엔드포인트로 분리하고 내부 구현은 공유 권장. 근거: 두 화면이 독립적으로 존재하며 UX 진입점이 다름.
