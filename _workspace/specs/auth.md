## 인증 토큰 체계 (JWT) — vault PIN과의 경계 정의
Sortmate 인증은 **2계층**으로 구성된다. 두 계층을 혼동하면 통합 시 토큰 충돌이 발생하므로 여기서 확정한다.

1. **1계층 — 계정 인증 (auth 모듈 소관, 본 명세):**
   - 로그인(소셜/이메일) 성공 시 서버가 **JWT accessToken + refreshToken** 발급.
   - `accessToken`: JWT, 만료 30분(`[가정]`), 모든 보호 API에 `Authorization: Bearer {accessToken}` 헤더로 전달.
   - `refreshToken`: 만료 14일(`[가정]`), accessToken 재발급에만 사용. 갱신 엔드포인트로만 교환.
   - JWT 클레임(`[가정]`): `sub`(userId), `email`, `provider`, `iat`, `exp`.
2. **2계층 — 앱 잠금 PIN (vault 모듈 소관, 본 명세 제외):**
   - PIN은 **로그인된 기기에서 vault 접근을 다시 잠그는 로컬 2차 인증**이며, JWT를 대체하지 않는다.
   - PIN 검증 결과는 JWT를 재발급하지 않고 클라이언트 세션의 vault-unlock 플래그로만 관리한다(`[가정]` — vault 모듈 확정 대상).
   - **경계 규칙:** auth 계약의 어떤 엔드포인트도 PIN을 요청/응답 필드로 포함하지 않는다. auth는 계정 소유권만 검증하고, PIN은 vault가 별도 검증한다.

## 사용자 흐름
```
[로그인 /login]
  ├─ 카카오/구글/애플 버튼 → 소셜 OAuth 인증 → (성공) 홈/보관함  ※홈은 auth 모듈 밖
  ├─ "이메일로 로그인" 버튼 → [이메일 로그인 폼]  ※폼 화면 미제공 [가정]
  └─ (비밀번호 분실 진입) → [계정 복구 /recovery] 또는 [비밀번호 재설정 /password/reset]

[계정 복구 /recovery (com_008)]
  ├─ "이메일로 인증" → 매직 링크 발송 → (토스트) → 이메일 링크 클릭 → [새 비밀번호 설정]
  ├─ "복구 코드로 인증" → 24자리 백업 키 입력 화면 → (검증 성공) → [새 비밀번호 설정]
  ├─ "고객 센터 문의" → 고객 지원 포털  ※외부/off-module
  └─ 뒤로가기(arrow_back) → 이전 화면(history.back)

[비밀번호 재설정 /password/reset]
  ├─ 이메일 입력 → "재설정 링크 발송" → (로딩→성공 토스트) 이메일로 링크 발송
  └─ "로그인으로 돌아가기" → [로그인 /login]

[새 비밀번호 설정 /password/new (com_007)]
  ├─ 진입 조건: 유효한 재설정/복구 토큰 보유 (링크 클릭 또는 복구코드 검증 통과)
  ├─ 진행 표시: 3단계 중 3번째 활성 → (복구방법선택 → 인증 → 새 비번설정) 흐름 암시 [가정]
  ├─ 새 비밀번호 + 확인 입력 → "비밀번호 변경" → (성공 토스트) → [로그인]으로 복귀 [가정]
  └─ 뒤로(arrow_back) → history.back
```

## 화면별 상세

### login_com_003_2 — 로그인
- **표시 데이터:** 브랜드명("Sortmate"), 슬로건 텍스트, 서비스 약관/개인정보 처리방침 링크(정적).
- **입력 요소:** [카카오로 계속하기], [구글로 계속하기], [애플로 계속하기] 버튼 3종, [이메일로 로그인] 텍스트 버튼.
- **상태 변형:** 버튼 press 마이크로 인터랙션만 존재(정적 목업). 에러/로딩 상태는 화면 미표현 → 계약에서 표준 에러로 정의.
- **이동 경로:** 소셜 3종 → OAuth 인증 후 홈(off-module). 이메일로 로그인 → 이메일 로그인 폼(`[가정]`).

### reset_password — 비밀번호 재설정 요청
- **표시 데이터:** 안내 문구, 성공 토스트 문구("링크가 발송되었습니다! 편지함을 확인해 주세요.").
- **입력 요소:** 이메일 `input[type=email]`(id=email, placeholder "예: name@company.com"), [재설정 링크 발송] submit 버튼.
- **상태 변형:** 기본 / 로딩("전송 중...", 버튼 disabled + 스피너) / 성공("성공적으로 발송됨" + 토스트). 실패 상태는 화면 미표현.
- **이동 경로:** [로그인으로 돌아가기] → `login.html`(`/login`).
- **`[자율결정]`:** 존재하지 않는 이메일이어도 성공 토스트를 노출하는 것이 화면 동작(1.5초 후 무조건 성공). 계정 열거(enumeration) 방지를 위해 서버도 **항상 200 성공 봉투**를 반환하도록 계약 확정. 근거: 화면이 이메일 유효성에 따른 분기 UI를 전혀 제공하지 않음.

### set_new_password_com_007 — 새 비밀번호 설정
- **표시 데이터:** 진행 트래커(3단계, 3번째 활성), 보안 강도 라벨("강함")+게이지, 비밀번호 가이드라인 2줄.
- **입력 요소:** 새 비밀번호 `input[type=password]`(id=password, placeholder "최소 12자"), 비밀번호 확인 `input[type=password]`(id=confirm-password), 각 입력의 표시/숨김 토글 버튼, [비밀번호 변경] submit 버튼.
- **비밀번호 정책(화면 근거):** 최소 12자 / 최소 1개 대문자 + 1개 특수문자 / "123"·"password" 등 흔한 패턴 금지.
- **상태 변형:** 기본 / 성공 토스트("비밀번호가 성공적으로 업데이트되었습니다."). 강도 미터는 클라이언트 표시용. 불일치·정책 위반 에러는 화면 미표현 → 계약 에러 코드로 정의.
- **이동 경로:** 뒤로(history.back). 성공 후 이동 화면은 미명시 → `/login` 복귀로 `[가정]`.
- **진입 전제:** 유효한 `resetToken`(AUTH-04 링크) 또는 `recoveryToken`(AUTH-07 검증) 보유 상태.

### account_recovery_com_008 — 계정 복구 방법 선택
- **표시 데이터:** 안내 문구, 옵션 카드 2종(이메일 인증 "수신함으로 매직 링크를 보내드립니다" / 복구 코드 "24자리 백업 키를 입력하세요"), 하단 "고객 센터 문의", 빌드 태그("COM-008 • Sortmate Secure Auth v2.4"), 복구 성공 토스트("이메일로 복구 링크를 보냈습니다.").
- **입력 요소:** [이메일로 인증] 버튼, [복구 코드로 인증] 버튼, [고객 센터 문의] 버튼.
- **상태 변형:** 각 버튼 클릭 시 토스트. 자체 폼 없음(선택 후 다음 화면으로 위임).
- **이동 경로:** 이메일로 인증 → 매직 링크 발송(AUTH-06). 복구 코드로 인증 → 24자리 입력 화면(화면 미제공, `[가정]`) → 검증(AUTH-07) → 새 비밀번호 설정. 고객 센터 → 외부 포털. 뒤로 → history.back.

## 데이터 모델 초안
> 백엔드 참고용 초안. 최종 스키마는 backend-dev 재량.

- **User**
  - `id` (PK), `email` (unique), `displayName`, `provider` (enum: `EMAIL`|`KAKAO`|`GOOGLE`|`APPLE`), `providerId` (소셜 고유 ID, nullable), `passwordHash` (provider=EMAIL만), `createdAt`, `updatedAt`.
- **PasswordResetToken** (AUTH-04/05용)
  - `id`, `userId` (FK), `token` (URL-safe, 발송 링크에 포함), `expiresAt`, `usedAt` (nullable), `createdAt`. TTL `[가정]` 30분.
- **RecoveryCode** (AUTH-07용)
  - `id`, `userId` (FK), `codeHash` (24자리 백업 키 해시), `usedAt` (nullable), `createdAt`. 1회용. 24자리 형식 `[가정]` = 영숫자 24문자(4-4-4-4-4-4 그룹핑 표시).
- **RefreshToken** (AUTH-03용, `[가정]`)
  - `id`, `userId` (FK), `token`, `expiresAt`, `revokedAt` (nullable).

## 가정 및 상충
- `[가정] AUTH-02` 이메일 로그인 입력 폼 화면이 제공되지 않음. 요청 필드(email/password)는 표준 관행으로 확정하되 화면 확보 시 재검토.
- `[가정] AUTH-03` 토큰 갱신/로그아웃 화면은 없으나 JWT 체계상 필수 인프라로 포함.
- `[가정]` accessToken 30분 / refreshToken 14일 / resetToken 30분 TTL — 화면 근거 없음. 보안 기본값으로 설정.
- `[가정]` 복구 코드 "24자리" 형식을 영숫자 24문자로 해석(화면 문구 "24자리 백업 키"만 존재).
- `[가정]` set_new_password 성공 후 `/login`으로 복귀(화면에 다음 라우트 미명시).
- `[가정]` 진행 트래커 3단계 = 복구방법선택 → 인증 → 새 비번설정 흐름.
- `[자율결정] AUTH-04` 계정 존재 여부와 무관하게 항상 성공 봉투 반환(계정 열거 방지). 근거: reset_password 화면이 이메일 유효성 분기 UI를 전혀 갖지 않고 무조건 성공 처리함.
- `[자율결정]` reset_password(단순 이메일 링크)와 account_recovery의 "이메일로 인증"(매직 링크)이 기능적으로 중복. 두 화면 모두 유지하되 **동일 백엔드 엔드포인트(AUTH-04/06를 하나로)로 볼 여지**가 있음 → 진입 맥락이 다르므로(재설정 vs 복구 허브) 계약상 별도 엔드포인트로 분리하고 내부 구현은 공유 권장. 근거: 두 화면이 독립적으로 존재하며 UX 진입점이 다름.
