# 통합 QA — 경계면 교차 비교

## 왜 교차 비교인가

각 파트는 자기 기준으로는 "완성"이다. 버그는 파트 내부가 아니라 **경계면**에서 산다: 필드명이 한 글자 다르고, 한쪽은 배열인데 한쪽은 객체를 기대하고, 계약에 있는 엔드포인트가 실제론 경로가 다르다. 그래서 존재 확인이 아니라 양쪽을 동시에 열어 놓고 비교한다.

## 검증 체크리스트 (모듈 단위)

### 1. 계약 ↔ 백엔드
- [ ] 계약의 각 엔드포인트에 대응하는 `@GetMapping`/`@PostMapping` 등이 존재하고 경로·메서드가 일치
- [ ] 응답 DTO의 필드명/타입이 계약의 Response 스키마와 일치 (camelCase 포함)
- [ ] 계약의 에러 코드가 `ErrorCode` enum에 존재
- [ ] 요청 DTO의 검증 어노테이션이 계약의 제약(필수/최대 길이)과 일치

### 2. 계약 ↔ 프론트
- [ ] API 함수의 경로/메서드/파라미터가 계약과 일치
- [ ] 응답을 사용하는 코드가 접근하는 필드가 계약 Response에 실제로 존재
- [ ] 에러 코드별 분기(예: `VAULT_LOCKED` → PIN 화면)가 계약의 에러 목록과 일치

### 3. 프론트 내부
- [ ] 모든 화면이 라우트에 등록되고, 명세의 "사용자 흐름"대로 이동 링크가 연결됨 (죽은 링크 없음)
- [ ] mock 사용 위치가 완료 보고서 기록과 일치 (기록 없는 mock 없음)

### 4. 동적 검증 (가능할 때)
- [ ] `gradlew.bat bootRun`(백그라운드)으로 서버 기동 → 계약의 요청 예시를 curl로 전송 → 실제 응답 JSON의 shape을 계약과 비교
- [ ] 인증 필요 엔드포인트에 토큰 없이 요청 시 401 봉투 응답 확인
- 기동 실패 시 정적 비교만으로 진행하고 리포트에 생략 사유를 명시한다.

## 자주 나오는 경계면 버그 패턴

| 패턴 | 예시 |
|------|------|
| 필드명 불일치 | 백엔드 `createdAt` vs 프론트 `created_at` |
| 봉투 이중 해제/미해제 | 인터셉터가 `data`를 풀었는데 페이지에서 또 `res.data.data` 접근 |
| 컬렉션 shape 불일치 | 계약은 `{ items: [...] }`, 구현은 배열을 바로 반환 |
| 페이지네이션 기본값 차이 | 백엔드 0-base page vs 프론트 1-base |
| 경로 변수 vs 쿼리 불일치 | 계약 `/items/{id}` vs 구현 `/items?id=` |
| 날짜 포맷 | ISO 8601 vs epoch millis |
| 에러 시 봉투 이탈 | 예외가 핸들러를 안 거쳐 Spring 기본 에러 JSON으로 나감 |

## 버그 리포트 형식 — `_workspace/qa/{모듈}_report.md`

```markdown
# {모듈} QA 리포트 ({날짜})

## 요약: 통과 N / 실패 N / 보류 N

## 실패 항목
### QA-{번호}: {한 줄 요약}
- 담당: backend-dev | frontend-dev | spec-analyst(계약 자체 결함)
- 근거: {계약 파일의 해당 구절 인용}
- 실제: {파일 경로:라인} — {실제 코드가 하는 일}
- 기대: {계약대로라면 어때야 하는지}
- 재현: {curl 명령 또는 확인 절차, 해당 시}
```

리포트 작성 후 실패 항목을 담당 에이전트에게 전달한다. 수정 완료 통보를 받으면 **실패했던 항목만 재검증**하고 리포트를 갱신한다. 이전 리포트가 있으면 회귀 검증(이전 실패 항목 재확인)을 최우선으로 한다.
