# 기능 명세 & API 계약 작성

## 왜 이 형식인가

백엔드와 프론트가 병렬로 개발되므로, 두 팀이 공유하는 유일한 접점이 계약 문서다. 계약이 모호하면 각자 다르게 해석해 통합 시점에 터진다. 그래서 필드 하나까지 확정하는 것이 이 스킬의 목적이다.

## 화면 분석 절차

1. 모듈에 속한 각 화면 폴더의 `code.html`을 읽는다 (`screen.png`는 레이아웃 확인이 필요할 때만).
2. HTML에서 추출할 것:
   - **표시 데이터**: 텍스트/이미지/숫자로 렌더링되는 모든 동적 데이터 → 응답 필드 후보
   - **입력 요소**: input, 버튼, 토글, 선택 → 요청 필드/액션 후보
   - **상태 변형**: 빈 상태, 에러, 로딩, 한도 초과 등 화면이 표현하는 상태 → 에러 코드/분기 후보
   - **이동 경로**: 링크/버튼이 가리키는 다른 화면 → 라우트 흐름
3. 화면에 없는 것은 만들지 않는다. 단, 화면이 암시하는 필수 이면 기능(예: 목록 화면 → 목록 조회 API)은 포함한다.

## 기능 명세 형식 — `_workspace/specs/{모듈명}.md`

```markdown
# {모듈명} 기능 명세

## 화면 목록
| 화면 ID (폴더명) | 화면 이름 | 라우트(제안) |

## 사용자 흐름
{화면 간 이동을 화살표로. 예: 홈 → 검색 결과 → 아이템 상세}

## 기능 목록
| ID | 기능 | 관련 화면 | 비고 |
{ID는 모듈접두어-번호. 예: AUTH-01}

## 데이터 모델 초안
{엔티티명, 주요 필드, 관계. 백엔드가 참고할 초안이며 최종 스키마는 백엔드 재량}

## 가정 및 상충
{[가정] / [상충] 표시 항목을 모아서 나열}
```

## API 계약 형식 — `_workspace/contracts/{모듈명}.md`

엔드포인트마다 아래 블록을 반복한다:

```markdown
### {기능 ID}: {한 줄 설명}
- **Method/Path**: `GET /api/items?category={category}&page={page}`
- **인증**: 필요 / 불필요
- **Request Body**: (없으면 "없음")
​```json
{ "name": "string(필수, 최대 50자)", "pinned": "boolean(선택, 기본 false)" }
​```
- **Response 200**:
​```json
{ "success": true, "data": { "items": [ { "id": "number", "title": "string" } ], "page": "number", "totalPages": "number" } }
​```
- **에러**: `404 ITEM_NOT_FOUND`, `403 VAULT_LOCKED` (코드와 발생 조건)
```

규칙:
- 모든 응답은 공통 봉투 `{ success, data, error }`를 따른다 (`springboot-conventions` 스킬과 동일 규격).
- 필드 타입 표기에 필수/선택, 제약(최대 길이, 범위)을 함께 쓴다.
- 목록 API는 페이지네이션 방식(page/size, 기본값)을 반드시 명시한다.
- 날짜는 ISO 8601 문자열(`2026-07-19T09:00:00Z`)로 통일한다.

## 계약 수정 시

- 기존 계약 파일을 직접 수정하고, 파일 상단에 `> 변경: {날짜} {무엇을 왜}` 한 줄을 추가한다.
- 이미 구현이 진행된 모듈의 계약을 바꾸면 backend-dev와 frontend-dev 모두에게 알린다. 알리지 않은 계약 변경은 조용한 통합 버그가 된다.
